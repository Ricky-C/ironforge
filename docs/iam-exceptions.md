# IAM exceptions: services where Resource: "*" is required

Some AWS service actions don't support resource-level permissions per [AWS's service authorization reference](https://docs.aws.amazon.com/service-authorization/latest/reference/reference_policies_actions-resources-contextkeys.html). For these actions, IAM policies must use `Resource: "*"`. This document is the authoritative list of every Ironforge IAM policy where that exception applies and the AWS-imposed reason for it.

When `CLAUDE.md` says "No `Resource: "*"`", that's the default rule. This file is the explicit allowlist of exceptions.

## Adding an entry

When a new AWS-imposed `Resource: "*"` is unavoidable:

1. Confirm the limitation in AWS's service authorization reference (link the specific actions/resources page).
2. Add a row to the table below.
3. Add an inline code comment at the policy site that points here, e.g.:

   ```hcl
   # ce:GetCostAndUsage does not support resource-level permissions per
   # AWS service authorization reference. Resource: "*" is required.
   # See docs/iam-exceptions.md.
   ```

## Removing an entry

When AWS adds resource-level support to a previously-broad action:

1. Narrow the inline policy to specific resource ARNs.
2. Remove the inline code comment.
3. Delete the corresponding row below.

## CloudFront — extended rationale

The CloudFront cluster of actions has the largest `Resource: "*"` surface in the project, both at the boundary and in the per-Lambda identity policy. The reasoning is layered and worth documenting once rather than re-deriving:

**Why ARN scoping is impossible.** CloudFront ARNs are ID-based: `arn:aws:cloudfront::<account>:distribution/<distribution-id>`. The distribution ID is assigned by AWS at create time and is not predictable in advance. Even for actions that *could* in principle scope to a specific ID after creation (`UpdateDistribution`, `DeleteDistribution`, `TagResource`), per-service IAM policy generation would need either (a) a two-phase create (`CreateDistribution` → narrow the identity policy → use the narrow grant for subsequent calls), or (b) accept `Resource: "*"` for the full lifecycle. The two-phase approach adds substantial complexity for security gain that's already covered by other layers; (b) is the simpler choice.

**Why tag-condition scoping doesn't work as a substitute.** The natural alternative to ARN scoping is `aws:RequestTag/ironforge-managed=true` — accept any CloudFront resource as long as the request tags it correctly. This fails because CloudFront's tag-on-create support is uneven across actions: `CreateDistribution` accepts `Tags` in the request and the condition would work; `CreateOriginAccessControl` doesn't accept tags at create time and the condition would either block legitimate operations or require per-action conditional logic that's fragile to AWS API additions. The boundary's job is "broad cap that catches the bugs identity policies miss" — a fragile cap that breaks legitimate operations is worse than no cap at the boundary layer when identity policies handle the load-bearing constraint.

**Why the boundary's `Resource: "*"` is acceptable.** The per-Lambda identity policy on `run-terraform` is also `Resource: "*"` for CloudFront actions, for the same ID-based-ARN reason. A boundary tag-condition would catch nothing that the identity policy doesn't — both layers face the same AWS-imposed constraint. A tighter boundary would just create false friction without raising the security floor. (See ADR-006's general principle: defense in depth earns its place when the boundary catches a class of bugs the identity policy doesn't; in this case both layers have the same ceiling.)

**What still mitigates the surface.**

- The `run-terraform` Lambda execution role's trust policy permits only `lambda.amazonaws.com` as assumer. No human or other workload can directly invoke the role.
- `run-terraform` is invoked only via Step Functions, which is itself invoked only by the API Lambda (POST `/api/services`), which is itself behind Cognito JWT auth. Three layers of access control before any CloudFront action fires.
- The template manifest's `allowedResourceTypes` whitelist (PR-C.1) is the upstream gate: if a future template tries to use a CloudFront resource type not declared in its manifest, `template-renderer/iam-policy.ts`'s drift check throws `IronforgeUnknownResourceTypeError` at policy-generation time, not at apply time.

**When to revisit.** If AWS adds resource-level support to CloudFront actions OR adds uniform tag-on-create support across all CloudFront resources, narrowing becomes possible — re-evaluate then. Until then, the broad surface is an accepted constraint.
